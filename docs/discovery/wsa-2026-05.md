# Working-Set Analysis (WSA) — May 2026 cache-sizing curve

**Status:** SKELETON — table is template-only. Operator fills the right-hand columns from a SHELF-35 Belady-oracle replay against a 7-day rep-1 trace.

**Owner:** T9 of `analyst_report_validation_rc9_plan_de82494e.plan.md`.

## Why this doc exists

We size shelf-pool cache capacity by guessing today: 240 GiB felt right after the first cutover; 500 GiB feels right after the May 4 working-set-vs-NVMe analysis. **Neither number is grounded in actual hit-rate curves.** Meta CacheLib's WSA approach (paper: "The CacheLib Caching Engine: Design and Experiences at Scale", OSDI '20) replays a real production trace through a Belady oracle at multiple cache sizes, plotting hit-rate vs cache size to identify the cliff.

This doc is the result of that exercise for shelf, performed against rep-1's 7-day workload trace. It answers one question:

> **At which cache size per pod does the marginal hit-rate gain drop below 2 percentage points per doubling?**

Above the cliff, more capacity buys negligible hits — better to spend the budget on T8 NVMe compression (effective capacity multiplier) or scale-out (more pods, higher peer-fetch ratio). Below the cliff, more capacity is the cheapest latency win available.

## Verdict template (fill after table is populated)

> **Marginal gain at 500 GiB → 1000 GiB is X pp.**
>
> - If X ≥ 5 pp → schedule a 500 → 1000 GiB bump on shelf-pool in the next operator window. Cost: ~$200/mo extra gp3 per pod × 6 pods = +$1.2k/mo. Compare against the latency $-saved benefit.
> - If 2 pp ≤ X < 5 pp → marginal. Prefer T8 compression (estimated 1.6–2× effective capacity at near-zero CPU) over more raw NVMe.
> - If X < 2 pp → 500 GiB sits at the cliff. Defer the bump indefinitely; pursue T8 compression as the next-step capacity move, OR scale shelf-pool 6 → 8 pods (more peer-fetch coverage, lower hit-ratio variance).

## Confounders to disclose in the verdict

- **Cold-start tail.** A 7-day window includes Foyer cold-start periods after pod restarts. The Belady oracle assumes warm cache throughout — over-estimates hit ratio by 1–3 pp vs reality. Acceptable for relative-comparison purposes.
- **Workload drift.** Rep-1's workload changes month-to-month as DBT models / Power BI dashboards are added. Re-run this analysis quarterly, not annually.
- **Per-replica variance.** Rep-0's workload is 67× larger by query volume (workspace memory: rep-0 ≈ 5.7 M queries/7d vs rep-1's 85 K). The 500 GiB-cliff finding from rep-1 does NOT auto-apply to rep-0; if rep-0 is cut over to shelf later, re-run this analysis with rep-0 trace.
- **Belady is the upper bound.** Real LRU / S3-FIFO hit ratios are 5–15 pp below Belady. The cliff identification is still valid; the absolute hit-rate numbers are not.
