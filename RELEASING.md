# Releasing

This document is the source of truth for *how* Shelf ships. The roadmap
(`ROADMAP.md`) covers *what* ships and on what trajectory; the changelog
(`CHANGELOG.md`) captures what has already shipped.

## Versioning

Shelf follows [Semantic Versioning 2.0](https://semver.org/spec/v2.0.0.html).

- **Major** (`vX.0.0`) — breaking changes to the public S3 shim surface,
  the LODC write API, the metrics/labels surface, the on-disk format, or
  any other interface a downstream consumer reasonably relies on.
- **Minor** (`v1.X.0`) — backwards-compatible additions: new endpoints,
  new metrics, new chart values, new `shelfctl` flags.
- **Patch** (`v1.0.X`) — bug and security fixes only. No new public
  surface.

Pre-1.0 releases (`v0.x`) explicitly do *not* carry SemVer guarantees;
the v0.5 line is the v1.0 release candidate (see `ROADMAP.md`).

## Release branches

We cut release branches off `main` once the corresponding minor or major
line is locked down.

- `main` — active development.
- `release/v0.5` — soak / RC line for v0.5 → v1.0.
- `release/vX.Y` — long-lived branch for each shipped minor line; we
  back-port security and bug fixes here for as long as the line is
  supported.

A release tag is always cut from a release branch, never directly from
`main`. The tag is `vX.Y.Z` (no leading `release/` and no `v0` prefix
shenanigans).

## Cut process

1. **Branch.** If this is a new minor line, branch `release/vX.Y` off the
   commit on `main` that is being shipped.
2. **Bump.** Open a PR to the release branch updating Cargo workspace
   versions, the Helm chart `version`/`appVersion`, and `CHANGELOG.md`
   (move `Unreleased` content under the new version with a date).
3. **Review.** Standard CODEOWNERS review plus an explicit BDFL
   sign-off (Year 1) on the bump PR.
4. **Tag.** After merge, push the annotated tag `vX.Y.Z` on the release
   branch. Tag pushes trigger `.github/workflows/release.yml` (shipped
   on a sibling PR — `aamir306/shelf#2`), which produces all release
   artefacts described below.
5. **Announce.** Update GitHub Releases with the changelog excerpt and
   any operational notes (upgrade caveats, deprecations).

## Release artefacts

Every tagged release publishes the following:

- **Multi-arch container image** for the daemon:
  `ghcr.io/shelf-project/shelfd:vX.Y.Z`, with `linux/amd64` and
  `linux/arm64` manifests. Floating tags (`latest`, `vX`, `vX.Y`) are
  updated for stable releases only.
- **Helm chart as an OCI artefact**:
  `ghcr.io/shelf-project/charts/shelf:vX.Y.Z`. The chart `appVersion`
  matches the daemon tag.
- **`shelfctl` static binaries** for `linux/amd64`, `linux/arm64`,
  `darwin/amd64`, `darwin/arm64`, attached to the GitHub Release.
- **SPDX SBOM** generated by [Syft](https://github.com/anchore/syft) for
  the daemon image and each `shelfctl` binary.
- **SLSA provenance** generated by the GitHub Actions provenance
  generator for every published artefact.
- **Cosign keyless signatures** (Sigstore / Fulcio + Rekor) for the
  daemon image, the chart artefact, and the `shelfctl` binaries. Public
  verification commands are documented on the GitHub Release page.

Reproducibility: the release pipeline is deterministic; given the same
git tag and the same builder image, artefact digests should match. If
they do not, that is a release-pipeline bug and blocks the next release.

## Patch and security releases

Patch releases follow the same cut process on the relevant
`release/vX.Y` branch. Security releases additionally:

- Are coordinated with the security advisory flow described in
  `SECURITY.md`.
- Are tagged before the advisory is published, so users have working
  artefacts at the moment of disclosure.
- May skip the public RC step when warranted; the bump PR is still
  reviewed against `release/vX.Y` rather than landing directly.

## What this document does not cover

- The plumbing of `release.yml` itself. That is owned by the release
  pipeline and lives at `.github/workflows/release.yml` (sibling PR
  `aamir306/shelf#2`); changes there are reviewed there.
- Internal compute or registry costs. Operational details live in
  `runbooks/` and `docs/`.
